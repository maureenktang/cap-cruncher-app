# Global Brief for AI Coding Assistants

## 🎯 Project Overview
- **App Name:** Cap Cruncher
- **Tech Stack:** Next.js 15 (App Router), TypeScript, TailwindCSS, shadcn/ui, Supabase.
- **Package Manager:** Always use `npm`. Do NOT use pnpm or yarn (no pnpm lockfile exists; project uses `package-lock.json`).
- **Database:** PostgreSQL via Supabase, accessed with Drizzle ORM (`drizzle-orm` + `postgres` driver).

## 🛠️ Development Commands
- **Install Dependencies:** `npm install`
- **Run Local Server:** `npm run dev`
- **Build Project:** `npm run build`
- **Lint:** `npm run lint`
- **Generate migrations:** `npm run db:generate`
- **Run migrations:** `npm run db:migrate`
- **Push schema directly (dev only):** `npm run db:push`
- **Introspect existing DB into schema:** `npm run db:pull`
- **Drizzle Studio (DB GUI):** `npm run db:studio`

## 🧱 Repository Architecture
- `/app/` — Next.js App Router pages and layouts.
  - `/app/auth/` — Auth pages: login, sign-up, forgot-password, update-password, confirm, error, sign-up-success.
  - `/app/protected/` — Auth-gated pages (redirects to login if unauthenticated).
  - `/app/contracts/` — Contracts feature page (early stage, raw data dump).
- `/components/` — Reusable UI elements.
  - `/components/ui/` — shadcn/ui primitives: button, card, badge, input, label, checkbox, dropdown-menu.
  - `/components/tutorial/` — Supabase starter tutorial components (can be removed once app is built out).
- `/lib/` — Shared utilities, Supabase client setup, and Drizzle DB client.
  - `/lib/supabase/client.ts` — Browser-side Supabase client (auth only).
  - `/lib/supabase/server.ts` — Server-side Supabase client (auth only).
  - `/lib/supabase/middleware.ts` — Supabase session refresh for middleware.
  - `/lib/db/index.ts` — Drizzle client instance (use this for all data queries).
  - `/lib/db/schema.ts` — Drizzle schema definitions (source of truth for DB tables).
  - `/lib/utils.ts` — General utilities (`cn`, `hasEnvVars`).
- `/supabase/migrations/` — SQL migration files generated by Drizzle Kit.
- `middleware.ts` — Next.js middleware: refreshes Supabase auth session on every request.

## 🔐 Auth
- Auth is handled entirely by Supabase SSR (`@supabase/ssr`).
- Session is managed via cookies; `middleware.ts` refreshes it on each request.
- Protected routes live under `/app/protected/` and redirect to `/auth/login` if no session.
- Auth flow: sign-up → email confirmation (`/auth/confirm`) → logged in.

## 📋 Agent Rules
- **Always plan before coding.** Before writing any code, produce a written plan: list the files you will create or modify, describe what each change does, and identify any risks or open questions. Wait for user approval before proceeding.
- **One task at a time.** Complete and confirm each step before moving to the next. Do not batch unrelated changes into a single response.
- **Ask when uncertain.** If requirements are ambiguous or a decision has meaningful trade-offs, ask rather than assume.
- **Refactor long files.** If a file exceeds 300 lines, split it into smaller focused modules as part of the task. Do not leave files over 300 lines.
- **Suggest shared functionality.** If you notice logic that is duplicated or could be shared across files, flag it as a suggested refactor. Do not apply it silently — surface it to the user and wait for approval.
- **Report blockers immediately.** If something is missing (env var, schema, dependency), surface it before attempting a workaround.

## ⚖️ Coding Conventions & Rules
- **TypeScript:** Strict mode enabled. Never use `any`. Always infer types where obvious.
- **Components:** Functional components only. Use arrow function syntax (`const Component = () => {}`).
- **Styling:** Use Tailwind utility classes only. Do not write custom CSS stylesheets.
- **State Management:** Prioritize URL state. Use Zustand only for complex global state.
- **Data fetching:** Use the Drizzle client (`lib/db/index.ts`) for all data access. The Supabase client is for auth only.

## 🚀 Deployment & CI
- **Hosting:** Vercel, connected directly to this GitHub repository.
- **Production deploys:** Every push to `main` triggers an automatic Vercel deployment.
- **Preview deploys:** Every PR gets a unique Vercel preview URL automatically.
- **CI workflow:** `.github/workflows/ci.yml` runs lint and build on every push to `main` and every PR. A PR should pass CI before merging.
- **GitHub secrets required for CI:** `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_OR_ANON_KEY` (set under repo Settings → Secrets → Actions).
- **Recommended workflow:** Open a PR → CI passes → merge to `main` → Vercel deploys.

## ⚠️ Known Gaps / Not Yet Built
- Drizzle ORM is used for all data access. Never use the Supabase client for data queries.
- `/app/contracts/page.tsx` is a placeholder (raw JSON dump, no real UI).
- The players table on the home page is a placeholder — no real data or schema yet.
- No protected/authenticated routes yet — all pages are public.
